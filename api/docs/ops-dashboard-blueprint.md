# Simplified Ops Dashboard Blueprint

This blueprint defines the custom logistics admin dashboard that should sit on top of Fleetbase without exposing the full default Fleetbase dashboard complexity.

## Goal

Build a simpler operations dashboard for:

- orders
- drivers
- vehicles
- staff users
- roles and permissions
- day-to-day logistics operations

The dashboard should keep Fleetbase as the source of truth, but hide the low-level multi-step setup behind simpler workflows.

## Recommendation

Do **not** start with a separate repo.

Build this inside the current Fleetbase stack:

- frontend inside [`console/`](/Users/leslieaine/shopify/fleetbase/console)
- backend helper endpoints inside [`api/`](/Users/leslieaine/shopify/fleetbase/api)

## Why This Is The Right Path

### Existing console structure already supports modular engines

The Fleetbase console already mounts engines in:

- [`router.js`](/Users/leslieaine/shopify/fleetbase/console/app/router.js)

The router is auto-generated by:

- [`fleetbase-extensions-generator/index.js`](/Users/leslieaine/shopify/fleetbase/console/lib/fleetbase-extensions-generator/index.js)

The console already includes:

- `@fleetbase/fleetops-engine`
- `@fleetbase/iam-engine`

from:

- [`console/package.json`](/Users/leslieaine/shopify/fleetbase/console/package.json)

That means the cleanest implementation is a **custom local Fleetbase extension/engine**, not a separate app.

### A separate repo would create extra work

A new repo would mean rebuilding:

- auth/session handling
- company context
- role and permission loading
- API client wiring
- layouts/navigation
- access control

That is unnecessary because this stack already has those pieces working.

## Build Shape

### Frontend

Create a new custom engine for a focused logistics workspace.

Suggested route:

- `/ops`

Suggested engine package name:

- `@leslie/ops-dashboard-engine`

Suggested implementation pattern:

- local engine package installed into `console` as a file dependency
- discovered by the Fleetbase extension generator
- mounted automatically into the console

### Backend

Add a new set of orchestration endpoints in `api` that simplify Fleetbase’s multi-record workflows.

These endpoints should not replace Fleetbase models.

They should:

- validate simpler request payloads
- create/update the required Fleetbase records behind the scenes
- return easier dashboard-friendly JSON

## Key Product Principle

The dashboard should expose **tasks**, not raw entities first.

For example:

- “Create driver” instead of “create user, then company user, then assign role, then create driver”
- “Assign driver to order” instead of editing low-level records manually
- “Create dispatcher” instead of manually building user and permission relationships

## Composite Workflows To Hide Complexity

### Create Driver

Single form fields:

- full name
- phone
- email
- active status
- role preset
- optional vehicle
- optional notes/license details

Backend should perform:

1. create or update `User`
2. create or update `CompanyUser`
3. assign the Driver role
4. create or update `Driver`
5. optionally assign vehicle

### Create Staff User

Single form fields:

- full name
- phone
- email
- role preset
- active status

Backend should perform:

1. create or update `User`
2. create or update `CompanyUser`
3. assign preset role/policies

### Assign Order

Simple action:

- choose order
- choose driver
- assign

Backend should perform:

- validate company context
- check capacity
- assign driver
- update dispatch state

## Data Strategy

Do not create parallel source-of-truth tables for the major logistics entities.

Use Fleetbase’s existing models as the source of truth:

- users
- company users
- roles
- permissions
- drivers
- vehicles
- orders

The simplification should happen in:

- orchestration endpoints
- frontend forms
- role presets

## Permissions Strategy

Use presets by default.

Only show raw permission editing in an advanced view.

This keeps the dashboard approachable while preserving Fleetbase’s flexibility underneath.
